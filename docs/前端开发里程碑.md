## 二、第一阶段（已完成）

### 2.1 工程与基建

- Vue 3 + Vite + TypeScript 项目初始化
- 公共请求封装：`src/utils/request.ts`（统一响应 `code` / `msg` / `data` 解包、错误提示）
- 按业务模块拆分 API：`src/api/`（dashboard、equipment、workorder、maintain、spare、system 等）

### 2.2 布局与导航

- 后台基础 Layout、路由、菜单骨架
- 路由入口：`src/router/index.ts`（数据看板 + 5 个核心业务列表页）

### 2.3 页面与组件

- **公共样式与结构**：列表页统一为中后台常见结构（筛选区 + 表格 + 分页）
- **PageHeader** 组件抽取与复用
- **数据看板**（`/dashboard`）：KPI + ECharts（费用趋势、工单状态分布等），已接对应接口；失败时有本地兜底数据，便于演示
- **五个核心列表页第一版**：
  - 设备台账 `/equipment/list`
  - 工单列表 `/workorder/list`
  - 保养工单 `/maintain/order`
  - 备件管理 `/spare/list`
  - 用户管理 `/system/user`

### 2.4 列表页「主链路」接口化（本仓库已具备）

在「列表数据加载、查询/筛选、分页、总数、接口驱动结构」范围内：

- 上述列表页与看板均已走统一请求层，列表页具备：`onMounted` 拉数、`v-loading`、查询重置后回到第一页、分页与每页条数、`total` 使用服务端总数
- 部分页面在请求失败时使用本地 **FALLBACK** 示例行，避免白屏（后续可与产品约定是否改为空态 + 明确错误提示）

**备件管理说明**：「库存状态」为前端根据 `stockQty` / `warningQty` 计算的展示字段；原「按库存状态筛选」已从表单移除，避免与分页总数不一致；若产品需要按状态筛，需后端提供查询参数后再接。

## 三、第二阶段（进行中）

### 3.1 目标

- 从「纯列表 + 占位按钮」进入 **新增 / 编辑 / 详情 / 删除** 等 **CRUD 闭环**（弹窗或抽屉等形式）
- 与后端约定接口契约，进入 **可联调** 节奏（字段、枚举、分页参数对齐）

### 3.2 建议推进顺序（可按业务调整）

1. 选定 1～2 个高频模块先做完整 CRUD（例如设备或工单），沉淀表单校验、提交、刷新列表的通用模式
2. 其余列表页复用同一套交互与错误处理
3. 导出、批量操作等按优先级插队

## 四、尚未全面展开（待办）

| 类别 | 内容 |
|------|------|
| CRUD | 新增、编辑、删除、详情页或弹窗；与列表联动刷新 |
