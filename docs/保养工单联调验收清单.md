# 保养工单收口 - 联调验收清单（第1步）

## 目标
将保养工单的四大主动作（派工/执行/完成/关闭）、规则生成、模板依赖与角色权限在一次联调中闭环验证，确认前端流程与 `API_CONTRACT.md` 对齐。

## 验收前提
- 登录账号可切换以下角色组合：`ROLE_ADMIN`、`ROLE_MAINTAIN_MANAGER`、`ROLE_EXECUTOR`、`ROLE_VIEWER`。
- 保持后端已提供以下能力：
  - `GET /maintain/order/page`
  - `GET /maintain/order/{id}`
  - `PUT /maintain/order/{id}/assign`
  - `PUT /maintain/order/{id}/start`
  - `PUT /maintain/order/{id}/finish`
  - `PUT /maintain/order/{id}/close`
  - `POST /maintain/order`
  - `GET /maintain/template/page`
  - `GET /maintain/template/{id}`
  - `GET /maintain/rule/page`
  - `POST /maintain/rule/{id}/generate`

## 用例清单

### UC-01：列表检索与分页
- [ ] 进入 `/maintain/order`，能按关键字、状态、模板、日期区间筛选。
- [ ] 分页参数变化后，`total` 与分页器联动且不重复/空跳转。
- [ ] 数据为空时展示空态。

### UC-02：手动建单
- [ ] 创建时必填项校验：设备、模板、计划日期。
- [ ] 非派工建单：仅建单成功后回列表，状态应为待派工。
- [ ] 建单并派工：需选择执行人和计划执行时间，成功后状态应转为待执行或待执行相关态。
- [ ] 提交失败时页面给出错误提示（不应进入死循环）。

### UC-03：派工动作
- [ ] 列表行状态为 `待派工` 时，显示并可进入“派工”动作。
- [ ] 点击后弹窗选择执行人和计划执行时间、提交成功后返回详情并反映最新状态/时间线。
- [ ] 无执行权限角色（如只读）不能点击派工，并有禁用原因提示。

## 第1步执行指引（UC-01 ~ UC-03）

本次执行范围先聚焦：列表检索、建单、待派工派工动作。建议按如下顺序跑三条闭环：

### 1) UC-01（列表检索与分页）

- 前置：使用 `ROLE_ADMIN` 或 `ROLE_MAINTAIN_MANAGER` 登录，打开 `/maintain/order`。
- 验证动作：
  - 关键字输入框可触发列表刷新，`queryForm.pageNum` 重置为 `1`。
  - 状态下拉可筛选 `待派工 / 待执行 / 执行中 / 已完成 / 已关闭`。
  - 模板下拉来源于 `/maintain/template/page`，默认可见，切换后列表刷新。
  - 计划日期时间段切换后带上 `startDate/endDate` 拉取列表。
  - 分页切页/改每页条数后 `total` 与分页器一致，不出现空白重复页。
- 期望返回：
  - `GET /maintain/order/page` 成功返回 `{ list: [...], total }`。
  - 失败场景在非联调环境可能走本地示例；联调环境建议接口返回 500 时显示空态提示，避免白屏。

### 2) UC-02（手动建单）

- 进入页面后点击 `手动创建保养工单`，打开 `MaintainOrderCreateDialog`。
- 用例 A（普通建单）：
  - 勾选关闭 `建单并派工`。
  - 仅填设备、模板、计划日期、备注（可选）后提交。
  - 期望提交体（`POST /maintain/order`）：
  - `assigneeId/ plannedExecuteTime` 不应上送。
  - 成功后列表刷新到第一页，目标工单状态显示为 `待派工`。
- 用例 B（建单并派工）：
  - 勾选 `建单并派工`，补齐执行人 + 计划执行时间。
  - 期望提交体包含：
    - `assignNow: true`
    - `assigneeId`
    - `plannedExecuteTime`
  - 成功后刷新列表，新建工单进入待执行链路（通常为 `待执行`/可交由 `待派工` 后进入派工状态流，由后端决定）。
- 校验点：
  - `设备`、`模板`、`计划日期`为空应触发前端必填校验。
  - 提交失败应弹出错误提示，不出现重复提交。

### 3) UC-03（派工动作）

- 在列表找到状态为 `待派工` 的工单。
- 点击 `派工` 或列表主动作入口后进入详情（或列表行）内派工弹窗。
- 验证 `MaintainOrderDetailDialog` 里“派工”按钮状态：
  - 有 `assign` 权限时按钮可点，无需关闭原因。
  - 无权限时按钮置灰，悬浮提示为“当前角色无权限执行该操作”。
- 提交参数（`PUT /maintain/order/{id}/assign`）应含：
  - `assigneeId`（必填）
  - `plannedExecuteTime`（建议填写）
  - `remark`（可选）
- 提交成功验证：
  - 列表/详情状态更新，时间线新增一条派工/相关流转记录（至少含 `action/operatorName/operateTime`）。
  - 弹窗关闭后返回主列表且当前行按钮应不再显示 `派工`（进入下一状态主动作）。

### 执行记录更新（本批）

- UC-01：`待执行`
- UC-02：`待执行`
- UC-03：`待执行`

如需快速回填表单数据，可使用下列最小有效参数：
- `equipmentId`: 任意已有设备 ID（如 `101`）
- `templateId`: 任意可见模板 ID
- `planDate`: `YYYY-MM-DD`
- `assigneeId`: 任意在编人员 ID
- `plannedExecuteTime`: `YYYY-MM-DD HH:mm:ss`

### 4) UC-04（执行动作）

- 选中状态 `待执行` 的工单，进入详情弹窗并触发 `执行`。
- 验证：
  - 点击 `执行` 前，详情头部不应禁用该动作按钮（管理员/保养主管角色下）。
  - 表单 `plannedExecuteTime` 为必填；不填时应阻止提交。
  - `assigneeId` 变更为可选字段，提交失败时应返回错误提示。
- 提交请求（`PUT /maintain/order/{id}/start`）关注：
  - `assigneeId`（可空，视业务）
  - `plannedExecuteTime`（建议上送）
  - `remark`（可空）
- 成功预期：
  - 工单状态变为 `执行中`。
  - 详情时间线新增执行动作记录（`action/operatorName/operateTime`）。
  - 列表主动作从“执行”切换为“完成”。

### 5) UC-05（完成动作）

- 选中状态 `执行中` 的工单，点击 `完成` 进入完成弹窗。
- 验证模板检查项来源：
  - 先发起 `GET /maintain/template/{templateId}`，并回填模板固定项。
  - 再允许填写自定义项（空名称不允许提交）。
- 耗材与附件：
  - 耗材记录支持新增/删除，`spareId` 为必选、`qty > 0` 为必填。
  - 附件上传通过 `/maintain/attachment/upload` 入库文件，再提交完成时带上 `attachmentIds`。
- 提交参数（`PUT /maintain/order/{id}/finish`）关注：
  - `finishResult`、`finishTime`、`remark`（可选）
  - `checkItemRecords`（固定检查项数组，含 `itemName/result/remark`）
  - `customCheckItems`（自定义检查项数组）
  - `spareUsages`（`spareId/qty/remark`）
  - `attachmentIds`（数字数组）
- 成功预期：
  - 状态切到 `已完成`。
  - 详情页中“执行与完成记录”可见提交内容与时间线新增完成记录。
  - 关闭/执行按钮失效，列表动作列不再出现完成入口。

### 6) UC-06（关闭动作）

- 适用场景：列表行/详情状态为 `待派工` 或 `待执行`。
- 验证：
  - 详情页 `关闭` 按钮在这两种状态可点；其余状态应禁用或不显示。
  - `closeReasonType` 与 `closeReasonRemark` 为必填。
  - 提交前应做空值校验，不合法不发起请求。
- 提交请求（`PUT /maintain/order/{id}/close`）应包含：
  - `reasonType`（枚举：`无法执行`/`重复工单`/`设备停用`/`其他`）
  - `reasonRemark`（文本）
- 成功预期：
  - 状态变为 `已关闭`。
  - 时间线追加关闭记录；详情头部状态更新。
  - 关闭后不再出现执行/完成/派工动作入口。

### 本批执行建议顺序

1. 先执行 UC-04，把 1 条 `待执行` 工单完成到 `执行中`。  
2. 再执行 UC-05，复用同一工单完整走到 `已完成`。  
3. 另找一条 `待执行` 或 `待派工` 工单执行 UC-06，验证关闭链路。  
4. 进行 UC-07 的规则弹窗验收，确认分页、查询、增删改、启停与生成入口。  
5. 进行 UC-08 的模板联调验收，核对建单/完成链路模板数据源一致性。  
6. 将 UC-04~08 的状态写回结果表，确认是否阻塞后续场景。

### 7) UC-07（规则管理）

- 打开 `/maintain/order` 页面右上角的 `规则管理` 按钮。
- 验证弹窗与列表能力：
  - 首次打开加载 `pageNum=1/pageSize=10` 分页数据。
  - 关键字和状态筛选可刷新列表。
  - 分页翻页不丢页/不重复加载，`total` 与分页器联动。
- 验证 CRUD 与动作：
  - 点击 `新增规则` 可提交新规则，提交后列表刷新并包含新记录。
  - 点击 `编辑` 读取行数据回填表单，保存后列表反映更新。
  - 点击 `停用/启用` 成功后状态标签变化且保持当前筛选上下文。
  - 点击 `立即生成` 不报前端错误，调用成功后提示且可刷新列表（后端返回可见新记录按其定义）。
- 提交参数与接口核对（均为 `src/api/maintain.ts` 对齐）：
  - `POST /maintain/rule`
  - `PUT /maintain/rule/{id}`
  - `PUT /maintain/rule/{id}/enabled`
  - `POST /maintain/rule/{id}/generate`
- 成功预期：
  - 弹窗内提示文本清楚，关闭/重试后无悬挂状态。
  - 规则页面状态可读且与 `maintenance` 主链路互不影响。

### 8) UC-08（模板联调）

- 验证模板列表数据源统一来自专用模板接口：
  - `维护工单列表筛选中的模板下拉` → `GET /maintain/template/page`
  - `手动建单弹窗模板下拉` → `GET /maintain/template/page`
  - `规则管理弹窗模板下拉` → `GET /maintain/template/page`
- 验证模板详情数据源：
  - 完成动作中“模板检查项”点击时，发起 `GET /maintain/template/{id}`。
- 成功预期：
  - 三处模板来源一致（字段名不再混用字典兜底逻辑）。
  - 模板检查项在完成弹窗内先返回固定项，再允许新增自定义项。
- 额外校验：
  - 模板接口异常时仅该模块降级，列表与主链路其余动作不应被全局阻断（或需在本批验收中确认后端策略）。

### 9) UC-09（权限与状态边界）

- 依次以三种角色进入 `/maintain/order` 与详情弹窗，逐项核对按钮是否符合预期。
- 权限依据来源（本项目）：
  - `src/stores/auth.ts` 的 `ROLE_ACTIONS` 与 `getMaintainActionDisabledReason`。
  - 按钮/入口都应走这套判定，且文案一致为中文提示。
- 测试矩阵（状态：待派工 / 待执行 / 执行中 / 已完成 / 已关闭）：

| 角色 | 入口能力 | 待派工 | 待执行 | 执行中 | 已完成 | 已关闭 |
|---|---|---|---|---|---|---|
| ROLE_VIEWER | 查看 | 派工/执行/完成/关闭均禁用 | 同上 | 同上 | 同上 | 同上 |
| ROLE_EXECUTOR | 查看+执行+完成 | 派工禁用、关闭禁用 | 执行可点、完成禁用、关闭禁用 | 完成可点 | 全部禁用 | 全部禁用 |
| ROLE_MAINTAIN_MANAGER | 与管理员接近 | 派工可点 | 执行可点 | 完成可点 | 禁用关闭/主动作 | 全部禁用 |
| ROLE_ADMIN | 与管理员接近 | 派工可点 | 执行可点 | 完成可点 | 禁用关闭/主动作 | 全部禁用 |

- 验证点（与实现代码对应）：
  - 查看按钮在无权限时 `tooltip` 提示“当前角色无权限执行该操作”。
  - 状态不允许时提示“当前工单状态不允许该操作”。
  - “规则管理”与“创建工单”按钮也受权限控制（无权限不允许点击）。
  - 列表、详情、弹窗中的动作按钮行为一致（同一状态下同一角色结论一致）。
- 预期禁用规则参考：
  - `assign`：仅 `待派工`。
  - `execute`：仅 `待执行`。
  - `finish`：仅 `执行中`。
  - `close`：仅 `待派工` / `待执行`。

3. 切 `ROLE_MAINTAIN_MANAGER` 与 `ROLE_ADMIN`：
   - 具备列表、建单、规则、派工、执行、完成、关闭（状态匹配）能力。
   - 关闭仍仅在待派工/待执行时可见（或可点）并按文案提示状态约束。
